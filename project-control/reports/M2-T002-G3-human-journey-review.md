## 2. Rework verification (commit `6d9fbd4`)

**F1 (correlation id) — confirmed a TEST-ONLY fix; the component was correct all along.** `FailureState.tsx` `Meta` has always rendered `<code data-testid="correlation-id">` in `UnexpectedResponseState` when the client delivers a non-null id, and the client reads `X-Correlation-ID` through `boundedToken` (the fixture's 32-char id passes). The untouched, previously-passing `failures.spec.ts` real-harness test (lines 57–60) already proved the same render path end-to-end. The root cause was stub infidelity: `route.fulfill` bypassed the real harness middleware, and I verified `e2e/harness/fixture_api.py` (`build_app`) genuinely sends `expose_headers=["X-Correlation-ID"]` with `allow_origins=["http://127.0.0.1:3000", …]` — so adding exactly those two headers to the stub makes it a **faithful replay of what the browser receives from the real harness**, not a convenience hiding a client bug. The client's null-id behavior without the header was itself honest (renders nothing, invents nothing). Production caveat correctly unchanged: the deployed API has no CORS policy (M2-T001 D8 carry-forward, still tracked; B-001 blocks deploy).

**F2 (geometry copy) — confirmed.** The canonical sentence "Geometry of type ${type} is recorded for this lot from the official source — only recorded geometry is shown; a parcel outline is never drawn from assumptions." appears character-identically in `ConfirmScreen.tsx:204` (template) and `confirm-journey.spec.ts:37` (Point-instantiated, asserted verbatim). It is truthful — the F05 profile carries a real Point from the SODA `geom` column, and the sentence claims nothing beyond recording. The geometry-absent branch keeps its own honest copy. This is the only component copy change in the rework; grep confirms no third copy of either sentence exists.
