`git status` in the worktree shows ` M apps/web/.env.example` — a 5-line comment addition (documenting the M0-T015 sync:false declaration and build-time inlining) that is in NO commit. Repro: `cd .claude\worktrees\M0-T015 && git status --short && git diff apps/web/.env.example`. Required correction: orchestrator must deliberately commit it (with a one-line report amendment) or discard it — not leave the review target ambiguous.

**D-2 (MEDIUM, non-blocking here; required before B-002 provisioning): interpreter versions are not pinned.**
render.yaml sets no `PYTHON_VERSION` env var for nycdf-api and no `NODE_VERSION` for nycdf-web; `services/api` has no `.python-version` (verified: both greps exit 1). `pip install -r requirements.txt` never reads pyproject, so `requires-python = ">=3.12"` is not enforced at build, and Render's *default* interpreter version drifts over time — a hidden default that partially undermines the requirements.txt reproducibility rationale ("a Render rebuild months apart must not silently pull a different FastAPI" — true for packages, not for the interpreter). Producer's own known-limitation §6 (local 3.11 vs target 3.12) makes the gap visible but does not close it. Fix is one env-var declaration per service (or `.python-version`); it must land before the Blueprint is provisioned. Carry-forward recorded.

**D-3 (LOW, informational): test-count arithmetic in the producer report is off.**
Report claims "13 tests" added and "pre-existing suite was 143". Actual: 14 collected cases in `test_security_middleware.py` (parametrize expands to 4), pre-existing 142 (collected by me), total 156 — the report's headline total is correct, the components are not. No evidence impact; note for report hygiene.

**D-4 (LOW, informational): cron restoration note names no milestone ID.**
The worker restoration cites "M1 connector-framework / job-system work (PRD §22)"; the cron cites only "scheduled source-monitoring work … (PRD 14.1)" with no `M<n>` identifier (render.yaml:142–143). The binding is task-based ("the task that delivers each module MUST restore its block"), which is durable, but the packet-review instruction asked for owed milestones. Recommend the orchestrator attach the cron restoration duty to a named ledger task when the source-monitor work is contracted (producer's own recommended-next-task 1 already asks for this).
