# LOW-1 — Path containment for trusted checked-in registry references (recorded under D-002 §2)

**Status:** recorded backlog item (ledger task **M0-T025**, backlog). NOT a blocker for the first
parallel build wave. NOT implemented under D-002. Do not amend PR #97 again for this (owner directive
D-002 §2).

## Finding

The Owner Directive Compliance validator (`tools/validate_directive_compliance.py`) and the shared
resolver (`tools/directive_registry.py`) read each directive manifest's `requirements_file` and
`verification_file` and open the named file relative to the directive's directory. These fields are
**trusted, checked-in registry references** authored only by the orchestrator and reviewed at the
directive gate. There is currently no explicit *path-containment* assertion that the resolved file
stays inside the directive's own directory (e.g. a `"../somewhere/x.json"` or absolute path would be
opened as written).

## Why this is LOW, not a blocker

- The values are **not attacker-controlled input**: they are committed registry data produced by the
  orchestrator and independently reviewed before a directive becomes active.
- Behavior today is **fail-closed**: a missing/unreadable/mismatched file makes the directive
  unresolved and blocks acceptance. Nothing is silently trusted.
- No product-data, request path, or external input reaches these fields.

## Required disposition (preserved durably)

1. Add an explicit path-containment check so `requirements_file` / `verification_file` must resolve
   **within the directive's own directory** (reject `..` traversal and absolute paths), with a clear
   error.
2. **Preserve the current fail-closed behavior** — the containment check only narrows what is
   accepted; it never loosens it.
3. Positive tests (current in-directory refs still PASS) + negative tests (out-of-directory ref
   REJECTED).
