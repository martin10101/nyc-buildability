## 5. Deviations from the prior report's section-7 design (disclosed)

- **D-1 (defect found in the design): `if`/`then` conditionals are unusable here.** The design sketched `allOf` + `if/then`. `validate_contracts.py` enforces a keyword ALLOWLIST (`KNOWN_KEYWORDS`) that contains `anyOf`/`allOf`/`oneOf`/`const` but NOT `if`/`then`/`else` — an `if` keyword would fail the contracts CI job as "unknown/unsupported keyword". The conditionals are therefore encoded as `allOf: [ {anyOf: [{properties:{flag:{const:false}}}, {required:[...]}]} , ... ]`, which both the stdlib mini-validator and jsonschema enforce identically (cross-checked verdicts in the validator run below; the two new invalid fixtures prove rejection on both engines). Semantics are equivalent. The house rule "required must reference sibling-defined properties" was respected by keeping `required`-only branches free of `properties` siblings.
- Minor: three staleness contract fixtures were added (1 valid, 2 invalid) beyond the design's list, so the conditionals are proven at the fixture-CI layer, not only in pytest.
- Everything else follows the section-7 design as written (a–i), including the builder-side fresh-marker default, the additive cache accessor, dotted-path `VERSION_INTRODUCED`, the exemplar advance, and the two-file A1 client edit.

## 9. Assumptions and defaults

- Fresh-serve marker semantics: a `PlutoFetchResult` with `staleness is None` is by construction a fresh retrieval (the connector never sets it; only the resilience layer stamps cached serves), so the builder's fresh-marker default is truthful — including for test/e2e harnesses that call the connector directly.
- `no_match` cached results also receive a cache-hit staleness stamp; harmless and truthful (the route's 404 branch builds no profile). Not separately tested.
- `age_seconds` is a float from monotonic-clock subtraction; schema type `number` with `minimum: 0` is satisfied by construction (monotonic differences are non-negative).

## 10. Known limitations / not verifiable locally

- **No local Node toolchain (thin-client policy):** `tsc`, vitest, and the 43 Playwright web-e2e journeys run in CI only. Code-level reasoning: the generated union and the client array were widened together, so both directions of the `MutuallyEqual` exhaustiveness proof in `contract.ts` hold again, and the harness's real-builder 200s (now declaring 1.3.0) pass the client's version check because `1.3.0` is in the updated set. CI (`web-e2e`, `contracts-typegen`, `contracts-schema-bundle`, `api`, `contracts`) is the proof.
- The typegen generator intentionally does not emit the staleness cross-field conditionals into TypeScript (documented in the schema description and README); TS consumers rely on runtime validation, exactly as they already do for every other schema constraint (patterns, minimums).
- The `validate_contracts.py` output in section 7 is elided for length; the run is deterministic and reproducible.

## 11. Security / provenance impact

Positive, no new surface: staleness becomes a typed, machine-readable contract field instead of a notes-only convention (PRD principle 4 — stale data visibly labeled, now machine-checkably so: the schema REJECTS a payload claiming staleness without naming the upstream failure, or a cached serve hiding its age); the refreshed `correlation_id` description removes a falsifiable claim from the contract; `retrieved_at` provenance behavior unchanged (never rewritten to serve time); no secrets, no new endpoints, no new dependencies, no logging changes; `upstream_error_type` values are the connector's own typed classifiers, never raw upstream strings.
