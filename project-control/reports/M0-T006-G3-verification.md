# G3 Gate Report — M0-T006 "ADRs + Render Blueprint + deploy/rollback procedures"

**Reviewer:** code-reviewer (independent; producer was cloud-architect)
**Date:** 2026-07-15 | **Mode:** read-only per ADR-005 — report returned in-message; orchestrator recorded the gate.

## Verdict: FAIL

## Summary

The deliverable set is well-structured, honestly sourced, and largely meets its acceptance scenarios: the Blueprint contains zero literal secrets, alternatives are genuinely weighed against PRD 14.1, environment mapping is concrete, and every spot-checked platform claim carries an official URL with a retrieval date, with unverifiable items explicitly flagged instead of guessed. However, the failure-handling path (S4) contains an internal contradiction that is exactly what this scenario tests: ADR-003 and the runbook assert that a failed production migration "halts the pipeline automatically" and "blocks the dependent deploy jobs," while `render.yaml` in the same deliverable sets `autoDeploy: true` and ADR-002's promotion model has Render and Vercel deploying on the git push itself — platform-native triggers that do not wait for the GitHub Actions migration job. An operator following the incident runbook would hold a false belief about system state during a failed-migration incident. This is a single high-severity, small-to-fix defect; everything else is medium/low follow-up. The ADR-004 (drop Vercel) amendment is feasible without a rewrite and does not affect this verdict.

## Defects

1. **HIGH — Auto-deploy contradicts the migrations-first halt guarantee.** ADR-003 D2 (L22) states pushes to the tracked branch auto-deploy (`autoDeploy: true`, render.yaml L52/L90); ADR-002 promotion rule 3 (L53) says the `production` push "triggers: migrations… Render production deploy, Vercel production deploy." Yet ADR-003 failed-migration step 1 (L77) claims "a failed migration job blocks the dependent deploy jobs" and runbook §2.5 step 1 (L64-65) tells the incident operator "Pipeline halts automatically." Render auto-deploy and Vercel git-integration deploys trigger on the push, not on the Actions job — nothing shipped makes the halt true, and no source is cited for it. D4 (expand→deploy→contract) mitigates the schema-safety impact, but an incident runbook asserting a false system state is a high-severity hidden assumption. **Remediation (any one):** (a) gate Render deploys on CI checks — verify the current blueprint-spec `autoDeployTrigger: checksPass` (or equivalent) against https://render.com/docs/blueprint-spec and cite it; (b) set `autoDeploy: false` and trigger Render/Vercel deploys from the Actions pipeline after the migration job succeeds (deploy hooks/API); or (c) rewrite the halt claim as an explicit requirement on the future deploy workflow, state that platform auto-deploys are concurrent today, and name D4 as the actual safety property. Reconcile ADR-002 rule 3, ADR-003 step 1, and runbook §2.5 together.
2. **MEDIUM — Literal `ENVIRONMENT: production` with undefined staging override.** render.yaml L61-62 (repeated L94-95, L129-130) hard-codes `value: production`; the comment defers to "ADR-002 mapping," but ADR-002 defines no override mechanism, and ADR-002 L83 flags Blueprint-per-branch staging instantiation as unverified — a staging service deployed from this file would silently report `production`. Also a strict-letter deviation from S1's "env vars as references" (non-secret, so not gate-blocking alone). **Remediation:** move `ENVIRONMENT` to environment-scoped Render env groups (`fromGroup`) or `sync: false`, or define the concrete staging override in ADR-002.
3. **MEDIUM — Broken cross-reference: nonexistent "staging smoke-test cron."** ADR-002 L72 cites "the staging smoke-test cron (see `render.yaml` comments)" as the keep-warm mechanism preventing free Supabase project pausing; render.yaml contains no such cron or comment — only the prod source monitor. The pausing mitigation is therefore undefined. **Remediation:** add the cron (with cost note — no free cron tier) or rewrite the watch item to rely on development activity + the runbook unpause step only.
4. **LOW — Least-privilege: service-role key duplicated to GitHub environment secrets.** ADR-002 L44 places the Supabase service-role key + DB URL in GitHub environment secrets "for migration CI," but the cited Supabase CI flow (L46) needs only `SUPABASE_ACCESS_TOKEN`, `SUPABASE_DB_PASSWORD`, and the project ref. **Remediation:** remove the service-role key from CI secrets until a specific job demonstrably needs it.
5. **LOW — Runbook states unverified staging auto-deploy as fact.** Runbook §1.1 step 3 ("Render staging services… auto-deploy from `main`") presents behavior that ADR-002 L83 explicitly flags as unverified. **Remediation:** add `[confirm at first use]` or reference the ADR-002 flagged item.
6. **LOW — Mechanical YAML validation not executed in this gate.** The review sandbox denied Bash, so S1 syntax validation was a careful manual parse only. **[RESOLVED by orchestrator 2026-07-15: `yaml.safe_load` on render.yaml succeeded; three services web/worker/cron confirmed.]**

Per the gate rubric: Defect 1 (high, within scenario S4) drives the FAIL; defects 2-6 are recorded follow-ups and would not alone block.
