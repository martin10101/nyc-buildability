The project-control directory is authoritative. Producer agents may create only their own report files and must not run the control CLI. Reviewer agents are read-only: they return report content and a PASS/FAIL/BLOCKED verdict to the orchestrator and never run write-producing commands. All `tools/project_control.py` invocations (new-task, claim, progress, submit, gate, accept, checkpoint) are executed by the main-session orchestrator after validating the underlying evidence. Only the orchestrator may mark tasks accepted, change dependencies, modify the master plan, or unlock tasks. (Process decision ADR-005, 2026-07-14.)

Evidence-capture division of labor (owner directive, 2026-07-15): read-only reviewers produce reports and verdicts; when a reviewer's sandbox cannot execute commands (gh, python, network), the orchestrator captures the executable evidence into committed `project-control/reports/` artifacts and the reviewer verifies the stored evidence instead. A reviewer must not return BLOCKED solely because it cannot execute commands or write to the ledger — it requests orchestrator-captured evidence and verifies it. Gate results are always recorded by the orchestrator.

Agent-memory clarification (recorded 2026-07-16 after owner audit): ADR-005 is silent on agent memory. The governing permission is `docs/AGENT_OPERATING_SYSTEM.md` §7 ("Specialist agents may maintain concise project-scoped memory… Memory is not authoritative project status"), implemented via each agent's `memory: project` frontmatter. ALL agents — including read-only reviewers — may write ONLY under `.claude/agent-memory/<their-own-agent-name>/`. This is outside gate-evidence and ledger scope, never substitutes for a report, and reviewer read-only discipline applies to every other path.

Gate-verdict semantics (recorded 2026-07-16 after owner audit): the control CLI supports PASS/FAIL/BLOCKED only. A reviewer verdict of "PASS with required corrections" is recorded as PASS, and the corrections are BLOCKING for the next gate and for acceptance; the orchestrator records the blocking condition via `project_control.py progress --message` (never by hand-editing control files) and must not dispatch the next gate or accept until the corrections are applied, validated, and committed. Precedents: M1-T001 (C1–C6), M1-T003 (C1–C7), M1-T004 (C1–C9).

Report-preservation rule (recorded 2026-07-16 after owner audit): when a reviewer returns report content through the agent-return channel instead of writing a file, the orchestrator saves the return VERBATIM (transport entity-decoding only). If the orchestrator additionally produces a condensed transcription for readability, that file must say so explicitly and the verbatim original must be preserved alongside it. Never label a condensation "verbatim". Commits must stage exact intended paths — no broad `git add` of `.claude/agent-memory/` or whole directories.

## Directives registry (Owner Directive Compliance System, D-001)

`project-control/directives/` is a **provenance registry, not a lifecycle authority**. It records owner directives verbatim and maps them to where they are enforced; it never gates acceptance, unlocks tasks, or changes the master plan, and `tools/project_control.py` never reads `verification.json` to accept a task. All work-blocking decisions still flow through the CLI, blockers, holds, and gates (two-lane principle). The following rules apply under `project-control/directives/**`:

1. **IDs & layout.** Directive IDs are `D-<nnn>-<kebab-slug>` (mirroring blockers `B-nnn`); one directory per directive with `source-001.md` (verbatim, **IMMUTABLE** once active), append-only `source-00N-amendment.md`, `manifest.json`, `requirements.json`, `verification.json`. Requirement IDs are `D-<nnn>-R<nnn>`. Never name a directive file to collide with a `M<x>-T<n>` task id, and never place directive files in `blockers/`.
2. **Append-only after activation.** Corrections are **new amendment files** plus a `manifest.audit_log` entry — never edits to a committed `source-*.md`, never deletion/renumbering of an active requirement id. Source digests (`content_digest_sha256`) and `locked_requirement_ids` are enforced by `tools/validate_directive_compliance.py`.
3. **Multiple active directives.** `index.json` is a keyed registry; MULTIPLE directives may be `active` at once. States: `proposed | active | superseded | withdrawn | retired`. Supersession is explicit and symmetric (`supersedes`/`superseded_by`); retiring never deletes.
4. **Owner-gated activation.** AI may **draft** a directive but may not self-activate, amend, supersede, or withdraw an owner directive. Only the orchestrator writes directive records (same authority as the ledger; producers/reviewers remain read-only, writing only under `.claude/agent-memory/<self>/`). `verification.json` is written by an **independent verifier ≠ the producer** of `requirements.json`.
5. **Enforcement is regime-gated (migration).** Config `directive_compliance_regime` (`project-control/config.json`) sets version/effective date. A task is **in-regime** only when it carries `directive_regime_version` or `directive_refs`; legacy/pre-regime tasks are grandfathered so no in-flight work deadlocks. Regime entry is an explicit machine transition (`new-task`/`claim --directive-refs`), never `created_at` alone. Accepted/canceled tasks are immutable.
6. **A directive that must block work** does so through an existing mechanism (a `blockers/B-*.json` record scanned by `accept()`, an active hold rule, or task-packet fields) — never a directive-only field the CLI cannot see. Schemas live under `directives/schema/v1/`, additive/version-bump only; integrity is enforced by `tools/validate_directive_compliance.py` (wired into the control-plane CI), never folded into `project_control.py`.
