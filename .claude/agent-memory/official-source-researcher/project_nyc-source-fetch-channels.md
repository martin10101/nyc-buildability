Reliable vs blocked channels for NYC official-source research (observed 2026-07-14 through 2026-07-16):

- `www.nyc.gov` and `apps.nyc.gov/content-api/...` return HTTP 403 to non-browser sessions (bot protection) — page-level claims (download links, sizes, archive URLs) must go through a browser-capable capture or be flagged [NEEDS G1 RE-VERIFICATION]. The content-api pattern worked on 2026-07-14 (M0-T002) but was 403 on 2026-07-16 — availability is inconsistent, do not depend on it.
- `s-media.nyc.gov/agencies/dcp/assets/...` PDFs (READMEs, data dictionaries) fetch fine directly and are the authoritative DCP release documents.
- Socrata endpoints are dependable evidence: `data.cityofnewyork.us/api/views/<4x4>.json` for metadata (but summarizer-mediated timestamp readings can be wrong — read raw unix values), `resource/<4x4>.json?$limit=1` proves SODA liveness and gives a verbatim field sample. SODA omits null fields per record — never infer schema from record keys; use the api/views columns array.
- `dev.socrata.com/docs/*` and `nycplanning.github.io/*` fetch fine.
- ArcGIS REST is dependable evidence (observed 2026-07-16, M1-T003): `arcgis.com/sharing/rest/search?q=owner:DCP_GIS...&f=json` for item→service binding; `services5.arcgis.com/GfwWNkhOj9bNBqoJ/.../FeatureServer?f=pjson` and `/0?f=pjson` for schemas/CRS/editingInfo; `/query?returnCountOnly=true` for counts. Services carry NO version label — use `editingInfo.dataLastEditDate`. The full services directory JSON truncates in WebFetch (~850 services); probe specific service names instead.
- PDF technique: when WebFetch hits a PDF it can't parse (octet-stream or binary), it saves the file to the session tool-results dir; the Read tool CAN read it page-by-page IF the saved extension is .pdf (it refuses .bin). Socrata attachment API (`api/views/<id>/files/<assetId>?download=true&filename=x.pdf`) serves octet-stream (→ .bin, unreadable); the SAME DCP documents exist on `s-media.nyc.gov/.../bytes/<filename>.pdf` (naming held again: `zoningtaxlotdatabase_datadictionary.pdf`, `nyzd_metadata.pdf`) and save as readable .pdf. Prefer s-media; probing a hypothesized s-media filename live is fine (200 = verified, 404 = recorded).
- Socrata `blobby` datasets (file downloads, e.g. mm69-vrje) have createdAt=rowsUpdatedAt frozen at creation — freshness only via description version strings or content hash. Tabular `rowsUpdatedAt` IS meaningful and can expose real staleness (cross-check against `catalog.data.gov` harvest dates as a second signal).
- `zoningresolution.planning.nyc.gov` (alias `zr.planning.nyc.gov`) is bot-friendly (no 403s, observed 2026-07-16, M1-T004): server-rendered Drupal 9 on Pantheon, NOT a JS app; no JSON API (/jsonapi 404, ?_format=json 406, sitemap.xml 404, no Socrata ZR-text dataset). Legal text = clean HTML at /article-{roman}/chapter-{n}/{section} with machine-readable `<time datetime=...>` Last-Amended stamps and `<em>`-marked defined terms. Search param is `search_term` (NOT fulltext) — unknown params are silently ignored and return "0 results", so always run a positive-control query. Curl HEAD works on all its PDFs.
- WebFetch summarizer can transfer real URL fragments onto the wrong hrefs (applied the site's genuine `/index.php/` Drupal front-controller prefix — present in theme logo/footer links, 200 for HTML aliases, 404 for /sites/default/files/ static assets — onto zr-downloads PDF hrefs, 2026-07-16; corrected at G1). Any URL-level or verbatim claim must be confirmed via raw `curl` + grep of the saved HTML, never from summarizer output alone.
- `council.nyc.gov` and `legistar.council.nyc.gov` are automation-accessible official channels (no bot wall; verified at M1-T004 G1, 2026-07-16) — the official fallback for amendment/land-use ADOPTION records when nyc.gov 403s. The CPC report chain `a030-cpc.nyc.gov/html/cpc/report.aspx?num={ULURP}` 302s to a `www1.nyc.gov` PDF that is 403-bound (browser needed).
- ZR portal hidden metadata channel: `GET /ajax/get/amendment/section/{sectionEntityId}?_wrapper_format=drupal_ajax` + `X-Requested-With: XMLHttpRequest` → 200 JSON per-section amendment history (Effective Date | ULURP | Project Name | Action | Notes | Description). Section-entity IDs are a third namespace distinct from node IDs and section numbers. Undocumented — entity-print-grade stability caveats.

**Why:** Two research runs (M0-T002, M1-T001) hit these exact behaviors; M1-T001 run 1 was fully network-denied and unblocked via the fetch-plan → orchestrator-capture → stored-evidence pattern (ADR-005).

**How to apply:** When drafting fetch plans for the remaining PRD §8.1 source families, route claims to the channels above by reliability, pre-mark nyc.gov page claims as needing browser capture, and always request the raw api/views JSON values (not summarizer paraphrase) for timestamps.
